## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                 【Arduino UNO R3】                    【出力】
DHT11 ───────────────→┐                                     ┌──→ トランジスタ（低側スイッチ） → Fan Blade (V+ = 5V)
電源スイッチ（電源ON）─→│  判定処理・モード制御（自動 / 手動）   ├──→ 状態LED (D10/D11/D12: 電源=白(D10), 弱=緑(D11), 強=赤(D12))
手動ボタン（弱／強）────→┘                                    └──→ PWM出力（D9）
```

補足説明（図で表現しきれない場合）：
- DHT11は最短約2秒間隔のサンプリングのため、応答要件に注意。
- ユーザー指定によりセンサはDHT11を継続使用します。そのため、要件の「操作から1秒以内の応答」はセンサ読み取りに起因する自動制御部分で満たせません（DHT11の更新間隔 ≒ 2秒）。設計方針は以下の通りです：
  - DHT11を継続使用し、センサ由来の自動応答は約2秒の遅延を許容する。
  - ユーザー操作（ボタン押下）はセンサとは独立して即時反応させる（ボタン操作は1秒以内に反応する設計を維持）。
- トランジスタはArduinoのD9（PWM）でベースを駆動し、モーターは5Vへ接続してGND側をトランジスタでスイッチングする構成です。モーター端子にフライバックダイオード（例: 1N400x）を並列接続してください。ベースには1kΩ程度の抵抗と、浮遊防止のため100kΩ程度のプルダウンを推奨します。
- 手動ボタンは「弱／強」切替用で、押下で即座にPWM値を切替えます（手動が優先）。
- モーター駆動の目安：弱＝約60%（PWM=153）、強＝100%（PWM=255）。実測で調整してください。
- ヒステリシス：ON/OFFの判定に±1℃程度を入れることを推奨。

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（電源ON / モード切替）──→ [運転（弱）] ──（閾値超え／ボタン操作）──→ [運転（強）]
   ↑                                                         │
   └────────────────────────────（電源OFFで停止）─────────────┘

```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 待機中（自動） | DHT11を2秒間隔で読み、温度判定を待つ | 状態LED：短い点滅 |
| 待機中（手動） | 手動モードでボタン入力を待つ | 状態LED：常時点灯（電源ON時）または消灯 |
| 運転（弱） | モーターをPWM約60%で駆動 | 状態LED：短パルス点灯（弱を示す） |
| 運転（強） | モーターをPWM100%で駆動 | 状態LED：連続点灯（強を示す） |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

## 2. ソフトウェア設計

### 2-1. データ設計（使う変数・定数の洗い出し）

> ※ プログラムで扱う「情報の種類」を先に整理します。
> ソフトウェアワークシート（AI駆動開発）でやった「データの洗い出し」と同じです。

| 情報の名前 | 何を表すか | 型（イメージ） | バイト数 | 初期値 | 備考 |
|:--|:--|:--|:--|:--|:--|
| currentState | 現在の状態（0:待機 1:弱 2:強） | int | 2B | 0 | |
| sensorTemp | DHT11の温度（℃） | int | 2B | 0 | DHT11は約2秒間隔で更新。起動時は `sensorValid=false` で未取得を扱う |
| lastSensorMillis | センサー最終読取時刻 | unsigned long | 4B | 0 | millis()で管理 |
| lastDebounceMillis | ボタンデバウンス用タイムスタンプ | unsigned long | 4B | 0 | 50msデバウンスを想定 |
| modeManual | 手動モードフラグ（true=手動） | bool | 1B | false | |
| motorPWM | モーター出力（0〜255） | int | 2B | 0 | 弱=153(約60%)、強=255(100%) |
| sensorValid | センサー有効フラグ（true=有効） | bool | 1B | false | `readSensor()` 成功時に true にする。未取得時は安全動作（弱固定） |

**SRAM使用量（推定）**: 上記変数合計 約15B（UNOの2048Bに余裕あり）

### 2-2. 機能一覧・関数一覧（役割の定義）

> ※ プログラムを「処理の部品（関数）」に分割します。
> **ここではコードを書かず、「何をする関数か」だけ日本語で書いてください。**

| 機能ID | 機能名 | 関数名 | 担う「1つの仕事」 | 主な引数 | 戻り値 | 呼び出す場所 |
|:--|:--|:--|:--|:--|:--|:--|
| F01 | 初期化 | `setup()` | ピンモード設定・ライブラリ初期化・起動表示 | なし | なし | 起動時1回 |
| F02 | 待機・制御 | `loop()` | 状態管理・周期処理の呼び出し | なし | なし | 常時 |
| F03 | ボタン読出（デバウンス） | `readButtons()` | モード/風量/電源ボタンの状態を返す | なし | struct/flags | loop内 |
| F04 | センサー読出 | `readSensor()` | DHT11から温度を取得し`sensorTemp`を更新 | なし | int(℃) | 2秒毎 |
| F05 | 出力更新 | `updateOutput()` | `currentState`に応じてPWM/LEDを更新 | int state | なし | 状態変化時 |
| F06 | 電源起動（手動） | `handlePowerOn()` | 電源ONで手動起動処理 | なし | なし | ボタンイベント |
| F07 | 自動制御判定 | `autoControl()` | 温度閾値に基づき状態遷移を行う（ヒステリシスあり） | sensorTemp | int state | readSensor後 |
| F08 | LED表示制御 | `updateLED()` | 状態に応じた点滅/点灯パターンを実行 | state | なし | updateOutput内 |

### 2-3. タイミング設計（delay vs millis）

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| センサー読み取り | 約2000ms | millis | DHT11の仕様に合わせるため |
| ボタン監視 | 50ms以内の反応 | millis（デバウンス） | debounce処理で誤検知を防止 |
| LED・出力更新 | 状態変化時／500ms周期 | millis | ボタン監視と並行するため |

設計指針: `delay()` は使用せず `millis()` ベースの非ブロッキング実装を行うこと。

## 7. AIレビュー記録

### Q1: 設計の妥当性確認
**AIの回答（要約）：** 
- ピン競合: なし（D2=DHT11, D3=電源, D4/D5=ボタン, D9=PWM, D10=LED）。
- タイマー／ライブラリ衝突: D9/D10 は Timer1 を使うので Servo や Timer1 を使うライブラリを追加するとPWMに干渉します。
- DHT11応答: センサ更新 ≒2秒 → 要件の「1秒応答」は自動制御では満たせない。ボタン操作は即時処理可。
- 状態遷移の抜け: 起動時のセンサ未取得を扱う sensorValid（またはセンチネル値）を用意し、安全動作（弱固定等）を実装してください。
- delay() の問題: delay() は入力応答をブロックするので、millis() ベースの非ブロッキング実装を推奨。
- 電源注意: Arduino の 5V からモーター直給電は起動電流で電圧降下やリセットを招く可能性あり。外部電源やデカップリングを検討。
**対応した内容：** 
 - `sensorTemp` の初期化方針を明記：`sensorTemp` を起動時 `0`、`sensorValid` フラグ（初期 `false`）を追加。未取得時は安全動作（弱固定）を行う設計に変更。
 - 非ブロッキング実装を明記：`delay()` を避け `millis()` ベースで処理する旨を追記。
 - タイマー／ライブラリ衝突注記を追加：D9/D10 が Timer1 を使用するため `Servo` 等との干渉に注意と明記。
 - 電源・ノイズ対策を追加：モーター起動電流対策として外部電源推奨と、100µF＋0.1µF のデカップリングを推奨。
 - ボタン仕様を整理：ピン表の「短押し」表現を削除し、押下で切替と明確化。

### Q2: チャタリング・delay・ノイズの確認
**AIの回答（要約）：** 
- チャタリング対策は、lastDebounceMillis／デバウンス50ms、INPUT_PULLUP を使う方針が設計に明記されています（ソフトデバウンス）。
- 設計では delay() を避けて millis() ベースの非ブロッキング実装を明記しており、delay() の使用は問題になります（ボタン応答やLED更新がブロックされる）。
- ノイズ対策は必要です。
**対応した内容：** 
- ソフトデバウンスを明記：`lastDebounceMillis` と `debounceDelay = 50ms`、`INPUT_PULLUP` を使う方針を追記しました。
- `delay()` 回避を明記：全周期処理を `millis()` ベースの非ブロッキング実装とする旨を追記しました。
- DHT ライブラリの短時間ブロッキングに注意喚起：`readSensor()` とボタン処理を独立して動作させる設計にしました。
- ノイズ対策を追記：電源デカップリング（100µF＋0.1µF）、アナログ入力では RC ローパス／移動平均（4点程度）の併用を推奨する旨を明記しました。

### Q3: 要件への対応漏れ確認（任意）
**AIの回答（要約）：** 
- 手動応答時間の検証が無い（要件: 1秒以内、ただしDHT11は ≒2秒）

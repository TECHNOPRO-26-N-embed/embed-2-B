## 原稿

それではグループ2-Bから、私が担当した Mini扇風機システムについて発表します。
今日は、コードの構成と、なぜこういう形にしたのかを中心にお話しします。

**■ システムの全体構成**

まず、ファイル間の呼び出しは、上から順に大きく3つの層に分かれています。

一番上が、入り口の「`mini_fan_system.ino`」というファイル。
これは Arduino の仕様上どうしても C++ になりますが、中身はたった8行。
次の C 言語ファイルを呼び出すだけの、転送コードです。

真ん中が、研修の本番である「`app_main.c`」を中心とした C 言語の層です。
ここでシステム全体の状態を持って、機能モジュールを順番に呼び出します。

一番下が、Arduino の基本機能を呼ぶためのラッパー層「c_port」です。
こちらは後ほど、もう少し詳しくお話しします。

**■ 機能モジュール**

次に、機能ごとのCファイルです。全部で7つあります。

まず**ボタン入力**。50ミリ秒のデバウンス、つまり押した瞬間の振動を吸収する処理に加えて、電源を入れたときにボタンが押されたままだった場合の誤動作防止も入れています。

**つまみ**は、回転速度を0から255の範囲に変換しますが、小さな揺れに反応しすぎないよう、デッドバンドという仕組みも入れました。

**異常検知**は、つまみの線が外れて値が0や1023に張り付いた状態が1秒以上続いたとき、安全のためモーターを止めます。

**状態管理**は、OFF・運転中・異常停止という3つの状態を切り替えるところで、ここがシステムの脳みそにあたります。

**モーター制御**・**LED表示**・**デバッグ出力**も、それぞれ独立したファイルにしていて、1つのファイルが1つの仕事だけを持つように整理しています。

**■ なぜ C++ ファイルが2つ残るのか**

ここで「なぜ C++ のファイルが2つ残っているのか」を説明します。

研修テーマは C 言語でしたが、Arduino の世界では、`setup` と `loop` の入り口は必ず C++ から呼ばれる仕様になっています。
だから入り口の `.ino` ファイルだけは、どうしても C++ が必要です。

もう一つは、Serial 通信の部分です。
これも Arduino の「Serial」が C++ のクラスのため、C 言語から直接呼ぶことができません。
そのため `c_port_serial.cpp` という、たった20行の小さなファイルで橋渡しをしています。

逆に言うと、C++ はこの2か所、合わせて **30行未満**だけ。
それ以外の業務ロジック・判定・制御は、すべて C 言語で書いています。

**■ 研修目的との整合**

最後に、研修目的との関係です。

研修のテーマは「C 言語で組込み開発」でした。
本プロジェクトでは、書く対象・読む対象・テストする対象、そのすべてが、100パーセント C 言語のファイルに集約されています。

C++ が残っているのは、Arduino IDE の仕様上どうしても避けられない部分だけで、研修本来の目的は十分に達成できていると考えています。

以上で発表を終わります。ご清聴ありがとうございました。

## 発表後に質問が来た場合の備え（任意・暗記不要）

| 想定質問 | 答えのポイント |
|:--|:--|
| なぜ Arduino IDE をやめて完全 C にしないのか | PlatformIO 等で可能だが、UART やボード書き込みを自前実装する必要があり、研修目的（C ロジックの設計力）に対してコスパが悪いため |
| デッドバンドはいくつにしたか | 3（A/D値の差分3未満は前回値を維持。詳細設計§3-3） |
| 異常判定の閾値はなぜ「0〜2」と「1021〜1023」なのか | A/D変換のノイズに配慮し、近傍2ポイントまで許容することで、本当に張り付いた場合のみ検知できる |
| C/C++ 混在で名前衝突は起きないか | 各ヘッダで `#ifdef __cplusplus extern "C"` ガードを使い、C++ からは C シンボル名で呼ぶ。これにより name mangling の差を吸収している |

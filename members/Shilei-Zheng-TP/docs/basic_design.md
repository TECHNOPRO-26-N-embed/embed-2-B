## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                             【Arduino UNO R3】              【出力】
タクトスイッチ ──D2(INPUT_PULLUP)→ ┐                      ┌─D9(PWM)→ PN2222 ─→ DCモーター
                                  │                      │           ↑
ポテンショメータ ──A0────────────→ │      状態管理         │        Diode Rectifier（並列接続）
                                  │      PWM制御         │
                                  │     デバウンス        ├─D5──────→ LED（緑：運転中）
                                  └──────────────────────┴─D6──────→ LED（赤：停止/異常）
```

補足説明（図で表現しきれない場合）：
- モーター駆動用の電源・GNDは Arduino の 5V/GND と共通化（グループ統一仕様）。
- 追加機能としてDHT11を後付けする場合は D7 を予約（配線とソフトは未実装）。

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
   ↓（setup()完了。stateはOFFで開始）
[OFF（初期）] ──（ボタン押下エッジ）──→ [運転中] ──（ボタン押下エッジ）──→ [OFF]
   ↑                                       │
   │                                       │（つまみ値で速度を常時更新）
   │                                       ↓
   └──（ボタン押下エッジ）──[異常停止] ←──（連続NG値検出）
```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| OFF（初期） | モーター停止（PWM=0）、ボタン押下エッジを監視。電源ON直後は必ずこの状態 | 赤LED：常時点灯／緑LED：消灯 |
| 運転中 | A0のつまみ値を毎ループ読み取り、0-255にマップしてD9へPWM出力。同時にボタン押下を監視 | 緑LED：常時点灯／赤LED：消灯／モーター：PWM駆動 |
| 異常停止 | モーター強制停止（PWM=0）、ボタン押下でOFFへ復帰可能 | 赤LED：250ms周期で点滅（OFFと区別）／緑LED：消灯 |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

> 本設計では、OFF・運転中・異常停止のすべてからボタン押下でOFFへ戻れる構造。電源を切らずに復帰可能。

## 2. ソフトウェア設計

### 2-1. データ設計（使う変数・定数の洗い出し）

> ※ プログラムで扱う「情報の種類」を先に整理します。
> ソフトウェアワークシート（AI駆動開発）でやった「データの洗い出し」と同じです。

| 情報の名前 | 何を表すか | 型（イメージ） | バイト数 | 初期値 | 備考 |
|:--|:--|:--|:--|:--|:--|
| `currentState` | 現在の状態（0:OFF 1:運転中 2:異常停止） | int | 2B | 0 | 起動時は必ず0=OFF（要件3-1③） |
| `potValue` | A0の読み取り値（0〜1023） | int | 2B | 0 | 毎ループ更新 |
| `motorSpeed` | モーターへのPWM出力値（0〜255） | int | 2B | 0 | potValueをmap()変換 |
| `lastButtonMillis` | ボタンデバウンス用タイマー | unsigned long | 4B | 0 | 50ms判定の基準 |
| `lastButtonState` | 前回確定したボタン状態 | bool | 1B | true (HIGH) | INPUT_PULLUPでHIGH初期 |
| `abnormalStartMillis` | 異常値（0近傍/1023近傍）の張り付き開始時刻 | unsigned long | 4B | 0 | 0=正常。FAULT_MS（1000ms）以上継続で異常確定（グループレビュー後に変更） |
| `abnormalCount` | デバッグ用：累積異常検知回数 | int | 2B | 0 | シリアル出力でFAULT発生履歴を観察するため |
| `lastLedToggleMillis` | 異常時LED点滅用タイマー | unsigned long | 4B | 0 | 250ms周期 |
| `ledBlinkState` | 赤LED点滅トグル | bool | 1B | false | 異常停止時のみ使用 |
| `lastDebugMillis` | シリアル出力レート制限 | unsigned long | 4B | 0 | 500ms周期 |

> [!CAUTION]
> **SRAM使用量チェック（Arduino UNO R3 の上限は 2048B）**
>
> グローバル変数の合計: **26 B**（グループレビュー後に `abnormalStartMillis` 4B を追加）
>
> | 合計バイト数 | 判定 |
> |:--|:--|
> | ～1000B | ✅ 余裕あり |
> | 1000～1500B | ⚠️ char配列の長さを見直す |
> | 1500B超 | 🔴 AIに相談し変数を整理すること |
>
> → **26B：✅ 余裕あり**

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| ボタンデバウンス | 50ms | **millis** | つまみ読取・PWM出力を止めないため。要件「ボタン誤検知50ms」対応 |
| つまみ読取＋PWM反映 | 毎ループ（即時） | なし | 要件3-4「つまみを回した瞬間に反応」を満たすため遅延ゼロ |
| 異常時LED点滅 | 250ms | **millis** | 並行動作維持しつつ「異常」を可視化（OFFの常時点灯と区別） |
| シリアルデバッグ出力 | 500ms | **millis** | ログ過多を防ぐ |
| DHT11読取（A02・将来） | 2000ms | **millis** | DHT11の最小読取間隔は2秒。実装時に追加 |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

> 本設計では `delay()` を一切使わず、すべて `millis()` ベースとする。

## 3. ハードウェア設計

### 3-1. 部品一覧とピン接続

> ※ デジタルピン: D2〜D13 / アナログピン: A0〜A5
> 特殊用途: D0/D1（シリアル通信・使用禁止）、A4/A5（I2C）、D10〜D13（SPI）

| 部品名 | 型番・種類 | Arduinoピン | 入力 / 出力 | 備考 |
|:--|:--|:--|:--|:--|
| タクトスイッチ | プッシュボタン | D2 | 入力 | INPUT_PULLUP使用、押下でLOW |
| ポテンショメータ | 10kΩ可変抵抗 | A0 | 入力 | 両端を5V/GND、ワイパーをA0 |
| DCモーター（共通部品） | 3-6V DCモーター | D9経由（PWM） | 出力 | PN2222のコレクタに接続。給電は5V/GND（グループ統一） |
| NPN Transistor | PN2222 | D9（PWM） | 出力 | ベースに1kΩ抵抗、エミッタGND、コレクタにモーター |
| Diode Rectifier | 1N4007相当（ELEGOOキット付属） | モーターと並列 | - | カソードを＋電源側に。逆起電力からトランジスタを保護 |
| LED（緑） | 5mm LED | D5 | 出力 | 220Ω直列、運転中に点灯 |

> **ピン競合チェック（必須）**
> - D0/D1 は Serial 通信専用のため使用禁止
> - I2C 機器（LCD等）は A4/A5 を使用
> - 同じピンに複数の部品を繋いでいないか確認

**ピン競合チェック結果：** 競合なし
- D0/D1 未使用
- デジタルは D2/D5/D6/D9（および予約D7）で 1ピン1部品
- アナログは A0 のみ
- I2C/SPI 未使用

### 3-2. 電気的考慮事項

> **【任意】** 余裕があれば記入してください。抵抗値はAIに計算させてOKです。

| 用途 | 判断・計算メモ |
|:--|:--|
| LED電流制限抵抗 | (5V − VF≈2.0V) ÷ 10mA ≒ 300Ω → 220Ω使用（実電流約13.6mAで許容範囲内） |
| トランジスタ ベース抵抗 | 5V ÷ 1kΩ = 5mA。hFE≈100で約500mAまで駆動可能、本DCモーター（数百mA）に十分 |
| ボタンのプルアップ | INPUT_PULLUP使用（理由：Arduino内蔵プルアップで十分、外付け抵抗不要） |
| Diode Rectifier（逆起電力保護） | モーターと並列、カソード＝＋電源側。コイル逆起電力からPN2222を保護 |
| センサーの動作電圧 | ポテンショメータ：5V分圧で問題なし／DHT11（A02）：3.3〜5V対応 |
| 共通GND | モーター駆動電源とArduinoのGNDを必ず共通化（グループ統一仕様） |

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| チャタリング（ボタン） | ボタンを押したとき | 前回入力から50ms未満の入力を検出 | デバウンス処理（`millis()`差分判定）でスキップ |
| ポテンショ値の異常 | 接触不良・断線時 | `analogRead()` が **0近傍（≤2）または 1023近傍（≥1021）** に張り付き、**1000ms 以上継続**（`abnormalStartMillis` で計測） | 「異常停止」状態へ遷移、モーター停止、赤LED点滅 |
| モーター過電流／逆起電力 | モーター起動・停止・回転中の負荷変動 | ハードウェア側（PN2222 + Diode Rectifier）で吸収 | ソフトはPWM上限255で十分。要件3-4「過電流を防ぐ」に対応 |
| 起動時の初期状態 | 電源ON直後 | グローバル初期値＋`setup()`で `currentState = OFF` を保証 | 必ずOFFで開始（要件3-1③） |
| **起動時ボタン押しっぱなし** | 電源ON時に既にボタンが押されている | `readButtonEdge()` 内の起動ガード（`buttonReady` フラグ）で、最初に一度 HIGH を確認するまで押下エッジを返さない | OFF 状態のまま、ユーザーが指を離して再度押下するまで運転を開始しない（要件3-1③の境界条件対応） |
| 異常停止からの復帰 | 異常状態でボタン押下 | `readButtonEdge()` で押下を検知 | OFF状態へ戻し、再度ボタン押下で運転可能 |
| 共通GND未配線 | 配線ミス時 | グループ共通仕様で事前防止 | 結合前にテスタで導通確認（グループ統一） |

## 7. AIレビュー記録

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
- ピン競合はありません（使用: D2/D5/D6/D9、予約: D7、アナログ: A0）。
- 現時点はArduino標準機能で実装でき、ライブラリ競合はありません（DHT11追加時のみDHT系ライブラリを導入）。
- 状態遷移（OFF→運転中→OFF、運転中→異常停止→OFF）は成立しており、復帰経路もあります。
- `delay()` は使わない設計なので、応答性の問題は起きにくいです。
- 注意点は2つあります。①「連続10回（約1秒）」は時間保証になっていない。②モーターをArduino 5V直給電すると起動電流で不安定になる場合がある。

**対応した内容：**
- 異常判定は「回数」だけでなく、`millis()` を使った時間基準でも管理する方針にした（例: 20ms周期で判定し、50回連続で約1秒）。
- 電源はまず現行構成で確認し、リセットや電圧降下が出る場合は外部5V電源＋共通GNDへ切替える方針を追記する。
- DHT11を追加実装する場合は、2秒周期の読取を守る前提で設計する。

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
- チャタリング対策は含まれています（50msデバウンス）。
- `delay()` は使わない方針なので、ボタン・PWM・LEDを並行で扱える設計です。
- アナログ入力（A0）はノイズ対策が必要です。特にモーター駆動中は値が揺れやすく、誤判定の原因になります。
- 0/1023はつまみ端でも出るため、「極値」だけで異常判定すると誤停止の可能性があります。

**対応した内容：**
- 異常判定は「0/1023に張り付き」だけでなく「一定時間ほぼ変化なし」を合わせて判定する方針にした。
- 風量更新には微小変化を無視するデッドバンド（例: ±2〜±4）を入れる方針にした。
- ハード側のノイズ対策として、A0-GND間コンデンサ（例: 0.1uF）と電源平滑（例: 47〜100uF）を検討項目に追加。

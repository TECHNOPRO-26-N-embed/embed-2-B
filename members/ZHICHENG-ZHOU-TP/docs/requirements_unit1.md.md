### 1-1. ブロック構成（入力 → 処理 → 出力）

記入例：
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

↓ 自分のシステムを書いてください
```
【入力】                 【Arduino UNO R3】            【出力】
開始時間設定ボタン ─→┐   モーター制御                      ┌──→ モーター（扇風機）
停止リセットボタン ─→┤  タイマー状態管理・判定処理 ├──→ LED（動作中/停止）
                     └─────────────────────────┘




```


- ボタン入力で運転時間を設定し、タイマー開始後は残り時間を内部でカウントする。
- 動作中はモーターをONにし、タイマー満了または停止ボタン押下でモーターをOFFにする。
- LEDは状態表示専用とし、動作中と停止中が判別できる点灯パターンを持たせる。

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

 ※ ガジェットが「どんな状態を持つか」を定義します。
 この設計が loop() の骨格になります。

記入例：
```
[電源ON  初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

↓ 自分の状態遷移を書いてください
```
[電源ON  初期化]
  ↓（初期化完了）
[待機中] ──（開始ボタン押下）──→ [動作中] ──（タイマー満了）──→ [停止完了]
   ↑                                                     │
   └──────────────（停止リセットボタン押下）──────────────┘


```

各状態の意味：

 状態名  この状態のとき、何をしているか  LEDやブザーなど出力の様子 
------
 待機中  開始時間設定ボタン入力を監視し、運転時間を確定する  LED消灯（または緑LEDゆっくり点滅）
 動作中  残り時間をカウントしながらモーターを駆動する  LED点灯（動作中表示） 
 停止完了  タイマー満了または停止リセット押下で出力を停止した状態を保持する  LED点滅（停止完了表示） 
 異常停止（任意）  ボタン異常入力や想定外状態を検出した場合に安全側で停止する  LED高速点滅（異常通知） 

 [!TIP]
 すべての状態から「抜け出せるルート」があるか確認してください。
 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-2. 機能一覧・関数一覧（役割の定義）

 ※ プログラムを「処理の部品（関数）」に分割します。
 ここではコードを書かず、「何をする関数か」だけ日本語で書いてください。
 具体的な実装は詳細設計書（detailed_design.md）で行います。

 [!TIP]
 単一責任の原則（1関数 = 1つの仕事）
 「機能名」欄に requirements.md の機能名（必須・追加）を書き、それを実現する関数名を対応させてください。
 入出力の読み取り（readButton 等）や制御（updateOutput 等）は複数機能から使われる共通処理として「共通」と書いてOKです。

 機能ID  機能名  関数名  担う「1つの仕事」  主な引数  戻り値  呼び出す場所 
--------------
 —  初期化  `setup()`  ピンモード設定・ライブラリ初期化・起動確認  なし  なし  起動時1回 
 —  （共通）待機・制御  `loop()`  状態に応じて各関数を呼び出すメインループ  なし  なし  常時 
 —  （共通）ボタン読出  `readButtons()`  2つのボタンをデバウンスして押下イベントを更新  なし  なし  loop()内 
 —  （共通）出力更新  `updateOutputs()`  状態に応じてモーターとLEDを更新  byte state  なし  loop()内 
 F01  必須機能①：運転時間設定  `handleTimeSetting()`  待機中に開始設定ボタンで時間を切替える  なし  なし  loop()内 
 F02  必須機能②：タイマー運転  `runTimerControl()`  動作中に残り時間を減算し満了で停止する  なし  なし  loop()内 
 F03  必須機能③：手動停止リセット  `handleStopReset()`  停止ボタンで即時停止し待機へ戻す  なし  なし  loop()内 
 A01  追加機能①：終了通知点滅  `blinkOnFinish()`  停止完了状態でLEDを一定回数点滅させる  なし  なし  停止完了時 
 A02  追加機能②：長押し初期化  `factoryResetByLongPress()`  停止ボタン長押しで設定時間を初期値へ戻す  なし  なし  待機中 

 [!CAUTION]
 loop() の中で全部書こうとしていませんか？
 機能ごとに関数を分けることで、グループレビューや単体テストがしやすくなります。

### 2-3. タイミング設計（delay vs millis）

 ※ `delay(N)` はN ms間、すべての処理が止まります。
 複数の処理を並行させたい場合は `millis()` を使います。

 処理  必要な周期  delay  millis  判断の理由 
--------
 ボタン入力監視  10ms  millis  チャタリング除去と操作遅延低減のため 
 残り時間の減算  1000ms  millis  他処理を止めずに1秒単位で更新するため 
 動作中LED点灯制御  常時（状態変化時）  millis  loop内で状態に応じて即時反映するため 
 停止完了LED点滅  500ms  millis  ボタン入力を止めずに点滅表示するため 

 [!CAUTION]
 `delay()` を使う場合、その間はボタン入力・タイマー更新・LED制御がすべて止まります。

## 4. 異常系・エラー処理の方針

 異常ケース  発生条件  検知方法  対応方針 
--------
 チャタリング  ボタンを押したとき  前回入力から50ms未満の入力を検出  デバウンス処理でスキップ 
 モーター逆起電力ノイズ  モーターONOFF切替時  LED誤点滅やリセット発生を監視  ダイオード追加・配線短縮・GND共通化 
 ボタン長押し誤検知  押下時間が境界付近のとき  長押し閾値前後の判定ログを確認  閾値にヒステリシスを設ける 
 モーターが停止しない  制御信号OFF後も回転が続く  停止命令後1秒以内に電流低下しない  異常停止状態へ遷移し、電源を遮断して再起動要求 
 タイマー値不正  残り時間が負値過大値になる  remainingSeconds の範囲外判定  初期値に戻して待機中へ遷移

## 7. AIレビュー記録

 グループレビューの前に必ず実施してください。

### Q1 設計の妥当性確認

 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

AIの回答（要約）：
- ピン競合はなし（D2, D3, D6, D9利用）。D0D1未使用で問題なし。
- 状態遷移は「停止リセットボタン」で待機へ戻るルートが確保されている。
- `delay()`常用は入力取りこぼしにつながるため、全処理を `millis()` 基準に統一するのが妥当。

対応した内容：
- ボタン監視・タイマー減算・LED点滅をすべて `millis()` ベースで設計した。
- ピン割り当てを D2D3D6D9 に固定し、設計書全体で統一した。

---

### Q2 チャタリング・delay・ノイズの確認

 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

AIの回答（要約）：
- チャタリング対策（30〜50msのデバウンス）は設計に含まれており妥当。
- `delay()` を使うと停止ボタンの即応性が下がるため採用しない方針が安全。
- モーター駆動時は逆起電力対策（ダイオード）が必須。

対応した内容：
- `readButtons()` でデバウンス処理を実施する前提を追記。
- モーター保護としてフライバックダイオードを 3-1 と 3-2 に明記した。

---

### Q3 要件への対応漏れ確認（任意）

 【任意】 トレーサビリティ表（Section 5）を書いた場合に実施してください。

 「要件定義書の必須機能と、この基本設計書のSection 5 を比較して、設計漏れや結合テスト計画の抜けを教えてください。」
